## 代理工作流程簡介

人工智慧的格局正從「基於聊天的 AI」轉向「代理 AI」。雖然標準的 LLM 互動涉及單一提示和回應，但代理工作流程允許 AI 進行迭代推理、使用工具並維護狀態來解決複雜的多步驟問題。

這場革命的核心在於一組特定的框架，它們提供了自主性所需的協調邏輯、記憶體管理和工具整合。本指南重點介紹了作為該領域基礎的主要 **代理工具**，後續章節將解釋如何實施和優化它們。

:::important[像戰術手冊一樣閱讀]
當您根據角色評估每個工具時，此列表最為有用：協調、測試、上下文管理、 UI 品質和模型控制。
:::

![aftermath](/images/posts/aftermath.webp)

2026 年，編碼的「手藝」已進入黑暗時代。我們正生活在 **混亂爆炸 (Slop Overflow)** 的時代，在這裡「感覺式編碼」取代了架構，AI 代理在終端機中爭論您甚至沒有編寫的程式碼行。

隨著手動編碼變成一種「劣勢」，前進的唯一方法是停止扮演勞工，開始扮演經理。為了保持領先地位，您必須學會如何協調機器。以下是七個開源專案，旨在幫助您塑造 AI 代理並構建高效的自動化管道。

:::note[編輯背景]
以下範例是根據策略性工具選項來闡述的，而不是一體適用的預設選項。請根據您的架構、成本結構和風險承受能力進行選擇。
:::

## 1. [Agency](https://github.com/msitarzewski/agency-agents)

### 自動化創業公司人力配置解決方案

在過去，成為全端開發人員意味著精通 CSS、資料庫和 DevOps。在 2026 年，這意味著聘用合適的代理。**Agency** 提供了一個函式庫，其中包含模仿科技創業公司中每個角色的開源代理範本。

* **運作方式：** 您部署專業化的人格角色—前端架構師、成長駭客和 Twitter 參與者—而不是單一的通用聊天機器人。
* **優勢：** 將這些範本整合到 Claude Code 等環境中，您可以從構想轉變為功能性產品，而無需手動實施每個部門的「個性」或邏輯。

:::tip[何時使用]
當您的瓶頸是角色定義，而不是原始模型能力時，請使用此模式。
:::

## 2. [Promptfoo](https://github.com/promptfoo/promptfoo)

### 提示的單元測試框架

當您的整個程式碼庫都是由自然語言生成的時，您的「程式碼」實際上是您的提示。**Promptfoo** (最近被 OpenAI 收購) 將提示視為需要嚴格測試的軟體模組。

* **模型基準測試：** 它允許您跨不同模型 (GPT-4、Claude 3.5、Gemini 1.5) 運行相同的提示，以查看哪個模型能為您的特定用例產生最高品質的結果。
* **紅隊測試：** 它具有自動化的「紅隊」攻擊，用於檢測 14 歲的 Discord 用戶是否能通過提示注入誘騙您的聊天機器人洩漏您的私有 API 金鑰。

:::important[生產環境防護欄]
將提示測試視為 CI 檢查。如果提示是程式碼，則評估就是您的測試套件。
:::

## 3. [MiroFish](https://github.com/666ghj/MiroFish)

![simulate_market](/images/posts/simulated_market.webp)

### 多代理預測引擎

失敗的代價昂貴；預測失敗是免費的。**MiroFish** 是一個多代理模擬引擎，可從金融趨勢和突發新聞中提取即時數據，以創建世界的「數位孿生」。

* **社交模擬：** 它啟動數百個獨立的代理，這些代理會「討論」並對數據做出反應，從而有效地創建一個微型的、不斷演變的社交網絡。
* **策略預測：** 您可以使用它在編寫任何程式碼之前，在宏觀層面上測試應用程式構想，並以驚人的準確性預測市場反應。

:::caution[模擬限制]
預測系統可能會放大錯誤的假設。在承諾路線圖決策之前，請使用實際用戶或市場信號驗證模擬結果。
:::

## 4. [Impeccable](https://github.com/pbakaus/impeccable)

### 消除「感覺式編碼」 UI 混亂的解藥

大多數 AI 生成的 UI 都會受到「紫色漸層症候群」的困擾——它們看起來很花哨，但功能卻雜亂無章。**Impeccable** 是一個基於 CLI 的工具，旨在通過 17 個專門的命令來完善前端設計。

* **精煉 (Distill) 命令：** 此命令會自動去除 LLM 經常引入的不必要複雜性，從而簡化您的 DOM 結構。
* **愉悅和動畫：** 一旦結構穩固，您就可以編程方式注入品牌顏色和微互動，以確保您的 UI 不會看起來像通用範本。

:::tip[實用工作流程]
先減少複雜性，再添加細節。團隊通常會按此順序獲得比風格優先生成更好的結果。
:::

## 5. [Open Viking](https://github.com/volcengine/OpenViking)

![refined_core](/images/posts/refined_core.webp)

### 代理的層級上下文管理

2026 年的金科玉律：**如果上下文是垃圾，輸出就是垃圾。** **Open Viking** 拒絕標準的「將所有內容塞入向量資料庫」的方法。相反，它直接將代理的記憶體、技能和資源組織到文件系統中。

* **層級載入：** 它使用分層系統，只為特定任務載入必要的上下文。這大大減少了令牌消耗，為您節省了 API 費用。
* **長期壓縮：** 它會自動精煉和壓縮代理的歷史記錄，這意味著代理在使用時間越長，實際上會變得越聰明 (且更便宜)。

:::important[成本控制]
上下文紀律通常是提高代理系統質量和降低成本的最快方法。
:::

## 6. [Heretic](https://github.com/p-e-w/heretic)

### 「非覺醒」模型解放器

大多數前沿模型都帶有「護欄」，可以阻止它們執行某些任務。**Heretic** 使用一種稱為 **消除 (obliteration)** 的技術來移除這些限制，而無需昂貴的後期訓練或微調。

* **運作方式：** 它針對負責 Google Gemma 等模型中「拒絕」行為的特定權重和激活。
* **結果：** 您將獲得一個服從所有命令的模型，而沒有「作為一個 AI 語言模型...」的講座。

:::warning[安全與合規]
移除安全限制可能會帶來法律、政策和安全風險。在任何實際環境部署之前，請使用嚴格的治理。
:::

## 7. [NanoChat](https://github.com/karpathy/nanoGPT)

### 100 美元的自訂 LLM 管道

如果您不信任大型科技公司，那就自己構建。**NanoChat** (靈感來自 Andrej Karpathy 的 NanoGPT) 從頭開始實現了整個 LLM 管道——包括令牌化、預訓練和微調。

* **主權：** 只需約 100 美元的 GPU 時間，您就可以訓練一個您擁有絕對控制權的小型語言模型 (SLM)。
* **教育用途：** 這是從「AI 用戶」轉變為「AI 架構師」的最佳方式，通過理解 transformers 的底層數學原理。

:::note[購買還是自建]
訓練自己的小型模型可以提高控制力並加速學習速度，但持續的維護和評估仍然需要嚴謹的工程。
:::
