## 1.0 引言：分散式系統不可迴避的法則

本系列的第一版，奠定了後端工程的角色、工具和初始架構模式的基礎。我們探索了單體式應用程式和微服務、SQL 和 NoSQL、REST 和 GraphQL。這些知識是建構功能性應用程式的必要基礎。然而，本卷將探討當我們擴展這些應用程式時會發生什麼；當我們的單一伺服器變成一個艦隊，我們的單一資料庫變成一個叢集，以及我們的進程內呼叫變成網路跳躍時。這是 **分散式系統** 的領域，它由一套不同且更嚴苛的規則所支配。

從單機系統過渡到分散式系統，並非複雜度的線性增加，而是一種典範轉移。在單機系統上成立的假設，例如可靠的網路、零延遲、即時操作，在分散式系統中將被打破。進階後端工程師的主要目標是建構能夠在底層環境固有的不可靠性下，正確且可靠地運作的系統。

### 1.1 分散式運算的八個謬誤

在 1990 年代，Sun Microsystems 的 L. Peter Deutsch 和其他人在 1990 年代匯編了一份謬誤清單；這是新手程式設計師在接觸分散式應用程式時，經常會犯的、並可能招致嚴重後果的假設。

:::caution[分散式運算的八個謬誤]

1. **網路是可靠的。** (事實並非如此。)
2. **延遲是零。** (事實並非如此。)
3. **頻寬是無限的。** (事實並非如此。)
4. **網路是安全的。** (事實並非如此。)
5. **拓撲不會改變。** (事實會改變。)
6. **只有一個管理員。** (事實上有許多。)
7. **傳輸成本是零。** (事實並非如此。)
8. **網路是同質的。** (事實並非如此。)

:::

本卷中討論的每一個模式、協定和架構，在某種程度上都是一種緩解這些謬誤後果的策略。

## 2.0 進階資料管理與一致性

在具有單一資料庫的單一伺服器應用程式中，資料一致性主要透過 ACID 交易來解決。在分散式系統中，一致性成為最困難的挑戰之一。

### 2.1 一致性光譜與 PACELC 定理

CAP 定理描述了網路分區期間的行為，而 **PACELC 定理** 提供了更完整的圖像：

:::note[PACELC 定理]
**「如果存在分區 (P)，分散式系統必須在可用性 (A) 和一致性 (C) 之間做出選擇。否則 (E)，當系統正常運行時，它必須在延遲 (L) 和一致性 (C) 之間做出選擇。」**
:::

這迫使進行細緻的架構討論。系統在故障期間可能為了可用性而犧牲一致性，但在正常運行時會優先考慮一致性而非延遲。

### 2.2 分散式交易：Saga 模式

兩階段提交 (Two-Phase Commit) 是同步的，不適合微服務。**Saga 模式** 透過本機交易和補償動作來管理跨服務的資料一致性。

:::tip[Saga 模式範例：電子商務訂單]

1. `訂單服務`：以 `待定` (PENDING) 狀態建立訂單，並發布 `訂單已建立` (ORDER_CREATED) 事件
2. `付款服務`：處理付款，成功時發布 `付款已處理` (PAYMENT_PROCESSED) 事件
3. `庫存服務`：更新庫存，成功時發布 `庫存已更新` (INVENTORY_UPDATED) 事件
4. `訂單服務`：將訂單更新為 `已確認` (CONFIRMED) 狀態

**故障處理：** 如果庫存服務失敗，付款服務將執行退款作為補償，訂單服務將取消訂單。
:::

**實作風格：**

:::note[Saga 實作方法]

* **協調 (Choreography)：** 服務發布/訂閱事件，沒有中央協調器
* **編排 (Orchestration)：** 中央協調器管理 Saga 狀態和補償性交易

:::

### 2.3 事件溯源 (Event Sourcing) 與 CQRS

這些模式建構高度可擴展且可稽核的系統。

* **事件溯源 (Event Sourcing)：** 儲存不可變事件，而非當前狀態。當前狀態是透過重播事件推導出來的。

:::note[事件溯源範例]

```json
// 與其儲存餘額：80
// 儲存事件序列：
[
  {"type": "AccountCreated", "initialBalance": 0},
  {"type": "DepositMade", "amount": 100},
  {"type": "WithdrawalMade", "amount": 20}
]
// 當前餘額 = 重播事件
```

:::

* **CQRS (Command Query Responsibility Segregation)：** 將寫入模型與讀取模型分離。

:::tip[CQRS 優點]

* 針對寫入和讀取優化的不同模型
* 命令端和查詢端的獨立擴展
* 透過獨立的上下文實現更好的領域模型

:::

### 2.4 後端工程師的資料庫內部知識

理解儲存引擎和複寫策略對於效能和可靠性至關重要。

:::note[MySQL 儲存引擎]

* **InnoDB：** 交易型、ACID 相容、適用於 OLTP 工作負載的列級鎖定
* **MyISAM：** 讀取速度快、表級鎖定、無交易 (新應用程式中已棄用)

:::

**複寫策略：**

:::tip[複寫模型]

* **主從複寫 (Leader-Follower)：** 所有寫入到主節點，讀取從複本進行 (最常見)
* **多主節點複寫 (Multi-Leader)：** 多個節點接受寫入，必須解決衝突複寫
* **無主節點複寫 (Leaderless, Cassandra style)：** 同時寫入多個節點，並以大多數節點成功作為讀取依據

:::

**交易隔離級別 (SQL)：**

:::note[SQL 隔離級別]

1. **未提交讀取 (Read Uncommitted)：** 可以讀取未提交的變更 (髒讀取)
2. **已提交讀取 (Read Committed)：** 只讀取已提交的變更 (可能發生不可重複讀取)
3. **可重複讀取 (Repeatable Read)：** 交易內的行值是一致的 (可能發生幻讀)
4. **序列化 (Serializable)：** 完全的序列化執行 (最高的一致性，最低的效能)

:::

## 3.0 彈性系統設計模式

彈性 (Resiliency) 是從故障中恢復並繼續運作的能力；是以優雅的方式處理故障，而不是完全阻止它們。

### 3.1 電路斷路器模式 (深入探討)

電路斷路器監控故障，並防止分散式系統中的級聯故障。

:::note[電路斷路器狀態]

* **關閉 (Closed)：** 正常運行，請求正常通過，監控故障
* **開啟 (Open)：** 對下游問題進行快速失敗 (fail fast)，在重試前超時
* **半開啟 (Half-Open)：** 使用單一探針請求測試下游恢復情況

:::

### 3.2 防波堤模式 (Bulkhead Pattern)

將應用程式元件隔離到不同的池中，以防止單一故障影響整個系統。

:::tip[防波堤實作]
為每個下游服務使用單獨的執行緒/連線池。緩慢的服務 A 不會影響服務 B 的池，從而防止系統完全故障。
:::

### 3.3 重試與超時模式

在分散式系統中處理暫時性故障的關鍵。

:::caution[重試最佳實踐]

* **超時 (Timeouts)：** 積極的超時可防止資源耗盡
* **指數退避 (Exponential Backoff)：** 增加重試間隔 (1 秒, 2 秒, 4 秒, 8 秒)
* **隨機抖動 (Jitter)：** 添加隨機性以防止驚群 (thundering herd) 問題

:::

### 3.4 速率限制 (Rate Limiting) 與負載卸載 (Load Shedding)

保護服務免受過載影響，並實施優雅降級。

:::note[速率限制策略]

* **令牌桶 (Token Bucket)：** 令牌會累積請求，使用時被移除
* **漏桶 (Leaky Bucket)：** 請求以固定速率處理，多餘的請求被丟棄
* **負載卸載 (Load Shedding)：** 在極端負載下拒絕低優先級的請求

:::

## 4.0 進階非同步通訊

非同步模式是彈性、鬆散耦合的分散式系統的基礎。

### 4.1 訊息佇列 (Message Brokers) vs. 事件日誌 (Event Logs)

具有不同權衡的訊息傳遞方法。

:::tip[訊息佇列特性]

* **RabbitMQ：** 智能路由、工作佇列、訊息佇列模型
* **Apache Kafka：** 事件流、持久日誌、多個消費者

:::

### 4.2 冪等性消費者 (Idempotent Consumers)

在訊息傳遞系統中處理「至少一次」傳遞的關鍵。

:::note[冪等性策略]

```text
function processMessage(message) {
  if (processedMessages.contains(message.id)) {
    return; // 跳過重複
  }

  // 處理訊息
  processBusinessLogic(message);

  // 標記為已處理 (與業務邏輯原子化)
  processedMessages.add(message.id);
}
```

:::

### 4.3 交易性發件箱模式 (Transactional Outbox Pattern)

在事件驅動系統中解決原子資料庫更新和事件發布的問題。

:::tip[交易性發件箱流程]

1. 在單一的本機交易中，更新業務實體 AND 將事件插入發件箱
2. 訊息轉發器非同步地發布事件並標記為已發送
3. 在不使用分散式交易的情況下，保證原子性
4. 提供「至少一次」傳遞語義

:::

## 5.0 大規模效能工程

識別和消除瓶頸的系統化紀律。

### 5.1 快取模式 (深入探討)

超越基本快取策略 (cache-aside) 的進階快取策略。

:::note[快取模式比較]

* **快取託管 (Cache-Aside)：** 應用程式碼管理快取，延遲載入
* **讀取穿透 (Read-Through)：** 快取處理從資料庫載入資料
* **寫入穿透 (Write-Through)：** 快取更新同步更新資料庫
* **寫入回寫 (Write-Back)：** 快取更新非同步刷新到資料庫

:::

**驚群問題緩解：**

:::caution[驚群問題 (Thundering Herd Problem)]
當快取項目過期時，數千個請求同時錯過快取並壓垮資料庫。解決方案：使用基於鎖的重新載入，其中只有第一個請求載入數據，而其他請求則等待。
:::

### 5.2 並行 (Concurrency) vs. 平行 (Parallelism)

效能優化的基本概念。

:::tip[工作負載匹配]

* **I/O 密集型工作負載：** 非同步模型 (Node.js, asyncio) 可以處理大量並行請求
* **CPU 密集型工作負載：** 平行處理 (Go, Java) 可以利用多核心

:::

### 5.3 分析 (Profiling) 與效能調校

無法測量，就無法優化。

:::note[效能分析]
使用分析工具來產生火焰圖 (flame graphs)，識別：

* 程式碼執行路徑中的 CPU 熱點
* 記憶體分配模式和洩漏
* I/O 瓶頸和等待時間

:::

## 6.0 進階 API 與安全性架構

用於管理分散式環境中複雜性的基礎設施級解決方案。

### 6.1 API Gateway 模式

管理客戶端與服務之間通訊的單一入口點。

:::tip[API Gateway 職責]

* **路由：** 將請求導向適當的微服務
* **身分驗證/授權：** 在邊緣驗證憑證
* **速率限制：** 執行使用政策和節流
* **請求轉換：** 調整請求以適應下游服務
* **可觀察性：** 集中式日誌記錄和監控

:::

### 6.2 服務網格 (Service Mesh)

用於安全、快速、可靠的服務間通訊的基礎設施層。

:::note[服務網格組件]

* **Sidecar Proxy：** (Envoy) 處理每個服務的所有進出流量
* **控制平面：** (Istio, Linkerd) 配置所有 sidecar proxy
* **功能：** mTLS、流量管理、分散式追蹤、可觀察性

:::

### 6.3 零信任安全 (Zero Trust Security)

分散式系統的「永不信任，始終驗證」安全模型。

:::caution[零信任原則]

* **基於身份的身分驗證：** 無論來源為何，驗證每個請求
* **最小權限存取：** 授予最低必需的權限
* **假定洩漏 (Assume Breach)：** 設計時預期內部可能遭受的侵害

:::

### 6.4 JWT (深入探討)：風險與緩解

理解 JWT 的漏洞和安全的實作。

:::caution[JWT 安全問題]

* **演算法混淆攻擊 (Algorithm Confusion Attacks)：** 欺騙伺服器使用弱演算法
  * *緩解：* 配置函式庫，僅接受強演算法 (RS256)
* **令牌撤銷 (Token Revocation)：** 無狀態令牌無法作廢
  * *緩解：* 在快速快取中維護一個撤銷黑名單

:::

## 7.0 結論：有原則的工程師

第二版系列深入探討了分散式系統工程。建構彈性、可擴展的後端系統需要對基本權衡有深入的理解：延遲與一致性、可用性與正確性、速度與安全性。

進階後端工程師會為失敗而設計，假定網路具有敵意，並應用 Saga、事件溯源、電路斷路器和服務網格等模式。最終的技能是推理複雜性；識別故障點、瓶頸和漏洞，並應用適當的緩解策略。
