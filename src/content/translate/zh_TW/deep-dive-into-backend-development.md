## 1.0 引言：超越 API 端點

在軟體開發的詞彙中，「後端」常被簡化定義為「伺服器上發生的事情」。這個定義雖然沒有錯，但卻極度不完整。它未能捕捉到建構現代應用程式數位基石所需的龐大複雜性、智力嚴謹性以及工程紀律。後端不僅僅是一段回應請求的程式碼；它是一個分散式系統、一個數據保管者、一個業務邏輯引擎，以及一座安全堡壘，所有這些都協同運作，以大規模、可靠地交付價值。

從軟體工程的角度來看，後端更應被視為一套基本原則的集合，而非技術的工具箱。我們的觀點是工程師和架構師的觀點：我們關注權衡、可擴展性、容錯能力以及複雜系統的長期可維護性。選擇程式語言或資料庫並非關乎受歡迎程度，而是基於需求、限制和特定問題領域的深思熟慮的工程決策。

### 1.1 後端作為系統的系統

現代後端很少是單一的、單體的應用程式。更準確的描述是，它是一個「系統的系統」，包含多個服務、資料庫、快取、訊息佇列和第三方整合。後端工程師的角色是設計、建構並協調這些組件，使其成為一個連貫、有彈性且高效的整體。這包括：

:::tip[後端核心職責]

- **資料建模與持久化：** 設計架構並選擇適當的儲存技術來表示應用程式的資料。
- **業務邏輯實作：** 將業務規則和流程轉化為穩健、可測試且易於維護的程式碼。
- **API 設計與管理：** 建立客戶端（前端、行動應用程式、其他服務）與系統互動的合約介面。
- **基礎設施與部署：** 管理系統在生產環境中運行所需的環境、配置和流程。
- **可觀測性與監控：** 儀表化系統以提供其健康狀況、效能和行為的可見性。
- **安全與合規：** 確保系統免受威脅並遵守相關的資料保護法規。

:::

## 2.0 基礎支柱

在我們能夠建構複雜的架構之前，我們必須掌握基本的材料。後端建立在三個支柱之上：計算環境（伺服器）、通訊協定（HTTP）以及資料交換的語言（序列化格式）。

### 2.1 伺服器：實體、虛擬與容器化

其核心是，後端是一個程式（或一組程式），運行在一台電腦上，稱為伺服器。伺服器技術的演進反映了持續朝向更高層級的抽象、效率和可管理性的驅動。

:::note[伺服器技術的演進]

- **裸機伺服器：** 專用於任務的實體機器。效能最大化，但昂貴且難以擴展。
- **虛擬機器 (VM)：** 虛擬化技術允許在一台實體機器上運行多個隔離的系統（例如 EC2、Compute Engine）。
- **容器：** 像 Docker 這樣輕量級的封包，用於打包應用程式及其依賴項。是現代部署的關鍵。

:::

### 2.2 HTTP 協定：網路的語言

超文本傳輸協定 (HTTP) 是驅動全球資訊網的應用層協定。對於後端工程師來說，理解其機制是必不可少的。

- **請求-回應模型：** HTTP 運作在一個簡單的模型上。客戶端向伺服器發送請求，伺服器則回傳回應。後端的主要職責是處理這些請求並制定適當的回應。
- **HTTP 請求的結構：**
  - **方法 (動詞)：** 指示要在資源上執行的期望動作。常見的方法包括：
    - `GET`：檢索資源。應是安全且冪等的。
    - `POST`：建立新資源。非冪等。
    - `PUT`：完全取代現有資源。應是冪等的。
    - `PATCH`：部分更新現有資源。不一定冪等。
    - `DELETE`：刪除資源。應是冪等的。
  - **URI (統一資源識別符)：** 指定請求所針對的資源（例如 `/api/v1/users/123`）。
  - **標頭 (Headers)：** 包含關於請求的元資料的鍵值對（例如 `Content-Type`、`Authorization`、`Accept`）。
  - **主體 (Body)：** 一個可選的載荷，包含資料，通常與 `POST`、`PUT` 和 `PATCH` 請求一起使用。
- **HTTP 回應的結構：**
  - **狀態碼 (Status Code)：** 一個三位數字代碼，指示請求的結果。這些代碼分為幾類：
    - `1xx`：資訊性
    - `2xx`：成功（例如 `200 OK`、`201 Created`）
    - `3xx`：重定向（例如 `301 Moved Permanently`）
    - `4xx`：客戶端錯誤（例如 `400 Bad Request`、`401 Unauthorized`、`404 Not Found`）
    - `5xx`：伺服器錯誤（例如 `500 Internal Server Error`、`503 Service Unavailable`）
  - **標頭 (Headers)：** 包含關於回應的元資料的鍵值對（例如 `Content-Type`、`Cache-Control`）。
  - **主體 (Body)：** 一個可選的載荷，包含請求的資源或錯誤資訊。
- **無狀態性 (Statelessness)：** HTTP 的核心原則是它是無狀態的。客戶端到伺服器的每個請求都必須包含理解和處理該請求所需的所有資訊。伺服器不會在請求之間儲存任何客戶端狀態。此設計對於網路的可擴展性至關重要。狀態通常在客戶端管理，或以 Token（如 JWT）的形式與每個請求一起傳遞。

### 2.3 資料序列化格式

當前端和後端進行通訊時，它們必須就交換資料的結構格式達成一致。這個過程稱為序列化。

:::note[JSON 範例]

```json {1,4-7}
{
  "userId": 123,
  "username": "testuser",
  "isActive": true,
  "roles": ["reader", "commenter"]
}
```

:::

- **XML (可擴展標記語言)：** JSON 的前身。它比 JSON 更冗長，可讀性較差。雖然在新的 Web API 中已被 JSON 大量取代，但在舊的企業系統、SOAP API 和某些組態檔案中仍然普遍存在。
- **Protocol Buffers (Protobuf)：** Google 開發的一種二進位序列化格式。它不可人類讀取。其主要優勢在於效能和效率。Protobuf 訊息比 JSON 更小，序列化/反序列化速度更快。它使用預定義的架構（`.proto` 檔案），強制執行服務之間的嚴格資料合約。這使其成為高效能、內部微服務通訊的絕佳選擇，尤其是在效能至關重要的情況下。

## 3.0 核心架構範式

後端系統的高層級結構就是其架構。選擇正確的架構是工程團隊可以做出的最重大的決策之一，因為它決定了系統將如何開發、部署、擴展和維護。

### 3.1 單體：統一系統

單體架構將應用程式建構成一個單一、統一的單元。所有業務邏輯、資料存取和 UI 服務組件都包含在單一程式碼庫中，並作為單一工件進行部署。

:::caution[單體缺點]

- **可擴展性挑戰：** 即使只有一個組件是瓶頸，也要擴展整個應用程式。
- **技術鎖定：** 從一開始就鎖定所選的堆疊。
- **靈活性不足：** 在沒有意外副作用的情況下難以修改。

:::

### 3.2 微服務：分散式方法

微服務架構將應用程式構建成一系列小型、自治的服務，每個服務都圍繞特定的業務能力進行組織。

:::tip[微服務優點]

- **獨立擴展：** 服務根據特定需求進行擴展。
- **技術自由：** 為每個服務選擇最佳工具。
- **容錯隔離：** 一個服務的失敗不會導致整個系統崩潰。

:::

### 3.3 無伺服器與 FaaS (函數即服務)

無伺服器是一種雲端執行模型，其中雲端供應商動態管理伺服器的分配和配置。開發者以函數的形式編寫程式碼，雲端供應商會根據事件執行它們。

:::note[無伺服器特性]

- 無需伺服器管理。
- 事件驅動執行。
- 按執行付費模式。
- 自動擴展和高可用性。

:::

### 3.4 選擇正確的架構：一切都關乎權衡

沒有「最佳」的架構。選擇取決於團隊規模、專案複雜性、可擴展性需求和開發速度。一種常見的、務實的方法是**從單體開始**，並隨著系統的成長和瓶頸的識別，策略性地拆分服務。這允許快速的初始開發，同時為未來在複雜性需要時遷移到微服務保留選項。

## 4.0 後端技術堆疊：原則性方法

技術堆疊是建構應用程式所使用的軟體組件的集合。選擇技術堆疊不僅僅是選擇流行的工具；而是做出符合系統需求和團隊專業知識的明智決策。

### 4.1 程式語言：關鍵選擇

程式語言的選擇對效能、開發者生產力和系統適於解決的問題類型有深遠的影響。

:::tip[語言比較]

- **Node.js (JavaScript/TypeScript)：** 由於其非阻塞事件循環，非常適合 I/O 密集型應用程式。
- **Python：** 簡潔易讀，擁有用於資料科學和快速開發的龐大生態系統。
- **Go：** 高效能、並行網路服務。簡潔的並行模型。
- **Java：** 穩健且平台獨立 (JVM)，龐大的企業生態系統。
- **C# (.NET)：** 強大的現代語言，具有用於企業用途的強大框架。

:::

### 4.2 框架：邏輯的腳手架

Web 框架提供了一組工具和函式庫，用於抽象化後端常見的任務（例如路由、請求處理、資料庫互動），使開發者能夠專注於應用程式特定的邏輯。

- **主觀性 vs. 非主觀性：**
  - **主觀性 (例如 Django、Ruby on Rails、Spring Boot)：** 這些框架為您做出了許多決策，並規定了構建應用程式的特定方式。它們提供了高生產力（「電池內建」），但在您需要偏離其慣例時可能受限。
  - **非主觀性 (例如 Flask、Express.js)：** 這些框架提供一個最小核心，並將大多數決策（例如資料庫層、模板引擎）留給開發者。它們提供了最大的靈活性，但需要更多的設定和決策。

### 4.3 資料庫：系統的記憶

資料庫可說是後端最關鍵的組件。它是應用程式的持久狀態。資料庫技術的選擇對系統的一致性、可擴展性以及它能有效支援的查詢類型具有長遠的影響。

#### 4.3.1 關聯式資料庫 (SQL)：結構與一致性

關聯式資料庫，使用結構化查詢語言 (SQL)，數十年來一直是業界標準。它們以具有預定義結構的表格來儲存資料。

:::note[ACID 屬性]

- **原子性：** 所有操作要麼全部成功，要麼全部失敗。
- **一致性：** 交易將資料庫從一個有效狀態帶到另一個狀態。
- **隔離性：** 並行交易不會互相干擾。
- **持久性：** 已提交的變更在發生故障後仍然存在。

:::

#### 4.3.2 NoSQL 資料庫：彈性與擴展

NoSQL 資料庫的出現是為了應對關聯式資料庫的限制，特別是針對大規模、高流量的資料（「大數據」）和需要靈活資料模型的應用程式。

- **BASE 屬性：** 與 ACID 相反，許多 NoSQL 資料庫提供 BASE 保證，它們優先考慮可用性而非嚴格的一致性。
  - **基本可用 (Basically Available)：** 系統保證可用。
  - **軟狀態 (Soft State)：** 系統的狀態可能會隨時間改變，即使沒有輸入。
  - **最終一致性 (Eventual Consistency)：** 當系統停止接收輸入後，最終會變得一致。
- **NoSQL 資料庫類型：**
  - **文件儲存 (例如 MongoDB、Couchbase)：** 以靈活的、類似 JSON 的文件格式儲存資料。非常適合架構不斷演變的應用程式。
  - **鍵值儲存 (例如 Redis、DynamoDB)：** 最簡單的模型。將資料儲存為鍵值對。對於簡單查詢非常快速。
  - **欄族儲存 (例如 Cassandra、HBase)：** 以欄而非列儲存資料。針對高寫入吞吐量和對大型資料集的查詢進行優化。
  - **圖形資料庫 (例如 Neo4j、Amazon Neptune)：** 設計用於儲存和查詢具有複雜關係的資料（例如社交網路、推薦引擎）。

:::caution[CAP 定理]
分散式資料儲存只能提供以下三者中的兩者：
**C**onsistency (一致性)、**A**vailability (可用性) 和 **P**artition Tolerance (分區容錯)。由於網路分區不可避免，權衡點在一致性與可用性之間。
:::

#### 4.3.3 ORM 與原生 SQL：抽象層的爭論

物件關聯對應器 (ORM) 是一個函式庫，它提供了一個抽象層，用於使用程式語言的物件和語法與關聯式資料庫進行互動。

- **ORM (例如 Django ORM、SQLAlchemy、Hibernate)：**
  - **優點：** 提高開發者生產力，資料庫無關的程式碼，降低 SQL 注入風險。
  - **缺點：** 可能產生效率低下的查詢，隱藏底層 SQL 的複雜性，執行複雜查詢可能很困難（「洩漏的抽象」）。
- **原生 SQL / 查詢建構器 (例如 SQLC、Knex.js)：**
  - **優點：** 對生成的 SQL 擁有完全的控制權以獲得最大的效能，更容易編寫複雜的查詢。
  - **缺點：** 冗長，與資料庫相關，如果處理不當，SQL 注入的風險較高。
- **務實的方法：** 對大多數簡單的 CRUD（建立、讀取、更新、刪除）操作使用 ORM，並為效能關鍵或高度複雜的查詢使用原生 SQL。

## 5.0 API 設計與建構

API 是定義不同軟體組件如何互動的合約。一個設計良好的 API 使用起來令人愉悅，易於理解，並且可以優雅地演進。一個設計不良的 API 則是持續混亂和錯誤的根源。

### 5.1 API 設計原則

:::tip[API 最佳實踐]

- **以資源為導向的設計：** 圍繞資源（名詞）進行結構化，使用 HTTP 方法來操作它們。
- **無狀態性：** 伺服器在請求之間不維護客戶端狀態。
- **冪等性：** 多次執行相同請求會產生相同結果。
- **集合使用複數名詞：** `/users` 用於集合，`/users/123` 用於特定用戶。

:::

### 5.2 REST (Representational State Transfer)

REST 是一種架構風格，而非正式協定。它利用 HTTP 的標準功能來創建 Web 服務。由於其簡潔性和與 Web 架構的對齊，它在 API 設計領域已佔據主導地位十多年。一個設計良好的 REST API 通常被描述為「RESTful」。

### 5.3 GraphQL

GraphQL 是 Facebook 開發的一種 API 查詢語言。它提供了一種比 REST 更高效、更靈活的替代方案。

- **GraphQL 解決的問題：** 使用 REST，客戶端經常面臨兩個問題：
  - **過度獲取 (Over-fetching)：** 客戶端下載的資料比其需要的多，因為端點返回固定資料結構。
  - **獲取不足 (Under-fetching)：** 客戶端需要發起多個到不同端點的請求，才能獲取其所需的所有資料。
- **GraphQL 的解決方案：** GraphQL API 暴露單一端點。客戶端發送一個指定其所需資料的查詢，伺服器則返回一個 JSON 物件，其中包含精確的所需資料，不多也不少。這使前端開發者能夠在單一次往返中獲取他們所需的資料。

:::note[GraphQL 查詢範例]

```graphql
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    email
    posts {
      id
      title
      content
    }
  }
}
```

:::

## 6.0 確保系統品質：非功能性需求

建構一個能正常工作的系統是一回事。建構一個能大規模可靠運行、在高負載下表現良好且能抵禦攻擊的系統，是另一種截然不同且更具挑戰性的工程問題。這些是非功能性需求，它們區分了穩健的系統和脆弱的系統。

### 6.1 可擴展性：處理增長

可擴展性是指系統透過增加資源來處理日益增長的工作量的能力。

:::tip[擴展策略]

- **垂直擴展：** 增加單一伺服器的資源（CPU、RAM）- 簡單但有限。
- **水平擴展：** 向資源池中添加更多伺服器 - 複雜但幾乎無限。
- **負載平衡：** 將流量分配到伺服器。
- **無狀態設計：** 對於會話數據使用外部共享儲存。

:::

### 6.2 效能與優化

效能即功能。緩慢的應用程式就是有缺陷的應用程式。

- **快取策略：** 快取是提高後端效能最有效的方法。它涉及儲存昂貴操作的結果，並在後續相同請求中重複使用它們。
  - **記憶體快取 (例如 Redis、Memcached)：** 一個外部、高速的資料儲存，用於快取經常訪問的資料（例如資料庫查詢結果、用戶會話）。Redis 因其多功能性（快取、訊息代理、佇列等）而被稱為後端的「瑞士軍刀」。
  - **內容分發網絡 (CDN)：** 一個地理上分散的代理伺服器網絡，將靜態資源（圖像、CSS、JS）緩存到靠近終端用戶的位置，從而顯著降低延遲。
  - **資料庫快取：** 大多數資料庫都有內建的快取機制來加速查詢執行。

:::note[非同步處理]

- **訊息佇列 (例如 RabbitMQ、SQS)：** 解耦服務並提高響應能力。
- **串流平台 (例如 Apache Kafka)：** 高吞吐量、即時資料處理。

:::

### 6.3 可靠性與容錯

系統會失敗。網路會分區。伺服器會崩潰。可靠性是指設計能夠承受這些故障並繼續運行的系統。

:::caution[容錯模式]

- **冗餘與高可用性：** 透過在不同位置運行多個實例來避免單點故障。
- **斷路器模式：** 監控故障並快速失敗，以防止級聯效應。
- **健康檢查：** 定期 ping 以偵測不健康的實例。
- **優雅降級：** 在組件失敗時提供降級功能。

:::

### 6.4 安全性：不可妥協的要求

安全性不是最後才添加的功能；它是從第一天就必須融入系統的根本屬性。

- **驗證 (Authentication) vs. 授權 (Authorization)：**
  - **驗證 (AuthN)：** 驗證用戶身份的過程。這通常透過用戶名/密碼、生物識別或社交登錄來完成。
  - **授權 (AuthZ)：** 確定已驗證用戶被允許執行的操作的過程。
- **常見安全協定：**
  - **OAuth 2.0：** 一個授權框架，允許第三方應用程式在不暴露憑證的情況下獲得對用戶在另一服務上帳戶的有限訪問權（例如「使用 Google 登錄」）。
  - **OpenID Connect (OIDC)：** 一個建立在 OAuth 2.0 之上的簡單身分識別層。它提供了標準化的驗證方法。
  - **JSON Web Tokens (JWT)：** 一種緊湊、URL 安全的方式來表示要在兩個實體之間傳輸的聲明。JWT 是一個簽名、無狀態的 Token，可以包含用戶身份和權限。它常用於在無狀態 API 中維護用戶會話。

:::caution[後端常見 OWASP 安全疑慮]

- 使用參數化查詢防止注入
- 在傳輸中 (HTTPS) 和靜態時加密資料
- 實施適當的存取控制
- 使用安全的依賴項和密碼管理

:::

## 7.0 現代開發與部署生命週期 (DevOps)

DevOps 是一套結合軟體開發 (Dev) 和 IT 營運 (Ops) 的實踐。它旨在縮短系統開發生命週期，並提供高軟體品質的持續交付。

:::note[DevOps 核心組件]

- **版本控制：** Git 用於程式碼和配置管理。
- **容器化：** Docker 用於可移植、一致的環境。
- **協調：** Kubernetes 用於自動化容器管理。
- **CI/CD 管道：** 用於建構、測試和部署的自動化工作流程。
- **基礎設施即程式碼：** Terraform/雲端模板用於配置。

:::

## 8.0 後端測試的藝術

全面的測試策略對於構建可靠的後端系統至關重要。

### 8.1 測試金字塔

用於Structuring your testing efforts 的模型。

:::tip[測試金字塔結構]

- **單元測試 (基礎)：** 孤立地測試單獨的函數/類別。快速、廉價，佔大多數測試。
- **整合測試 (中間)：** 一起測試多個組件（例如與真實資料庫）。
- **端對端測試 (頂部)：** 測試完整的用戶流程。緩慢、脆弱，謹慎使用。

:::

### 8.2 測試最佳實踐

:::note[額外的測試策略]

- **模擬/存根 (Mocking/Stubbing)：** 替換外部依賴項以隔離測試中的程式碼。
- **合約測試：** 確保 API 消費者/提供者遵守共享理解。
- **效能/負載測試：** 使用 k6 或 JMeter 等工具模擬高流量。

:::

## 9.0 結論：後端工程師不斷演變的角色

這次後端之旅帶我們從網路協定的基礎位元組到雲端原生架構的抽象高度。我們已經看到，後端開發不僅僅是編寫程式碼，而是設計、組合和管理複雜系統。它是一門關於權衡的學科：一致性 vs. 可用性、效能 vs. 成本、開發速度 vs. 操作穩定性。

今日的後端工程師是一位系統思考者、一個問題解決者和一個終身學習者。技術將不斷演進；無伺服器將成熟，AI/ML 模型將僅僅是另一個要整合的組件，新的架構模式將會出現。然而，我們討論的基礎原則；健全的架構、對非功能性需求的關注、穩健的測試和自動化部署；將繼續作為構建可靠且可擴展系統的持久基礎。最終目標不是掌握特定的框架，而是培養必要的工程判斷力，以選擇並運用正確的工具來應對數位世界複雜且不斷變化的挑戰。
